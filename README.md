# FAR-Trans Financial Asset Recommendation System

Course project for **DSAIT4050 — Information Retrieval** at TU Delft. The goal is to implement and compare recommendation models on the FAR-Trans dataset, a real-world financial transaction dataset from a European financial institution.

---

## Dataset

| Property | Value |
|----------|-------|
| Assets | 837 financial instruments (identified by ISIN) |
| Customers | ~32,000 |
| Transactions | ~388,000 buy/sell events |
| Source files | `data/raw/` |

**Raw files**

| File | Description |
|------|-------------|
| `customer_information.csv` | Customer profiles (type, risk level, investment capacity) |
| `asset_information.csv` | Asset metadata (category, sector, industry, market) |
| `transactions.csv` | Buy/sell events with timestamp, value, and channel |
| `close_prices.csv` | Daily closing prices per asset |
| `markets.csv` | Exchange and market metadata |
| `limit_prices.csv` | Min/max price bounds and profitability per asset |

## Preprocessing Pipeline

Run via `scripts/preprocess.py`. The pipeline has two main steps.

### Step 1 — Categorical Encoding

All string/category columns are encoded with `sklearn.LabelEncoder`. Missing values are filled with `"Unknown"` before encoding so no rows are dropped.

| DataFrame | Columns encoded |
|-----------|----------------|
| Customer | `customerType`, `riskLevel`, `investmentCapacity` |
| Asset | `assetCategory`, `assetSubCategory`, `sector`, `industry` |
| Transaction | `transactionType`, `channel` |
| Market | `country`, `marketClass` |

Encoder mappings are saved alongside the data so predictions can be decoded back to original labels.

**Deduplication** is also applied at this stage: both `customer_information` and `asset_information` may contain multiple time-stamped records per entity. Only the most recent record per customer/asset is kept.

### Step 2 — Temporal Dataset Variants

The FAR-Trans paper evaluates models across multiple points in time. This pipeline generates **30 independent variants**, each representing a different train/test split:

| Parameter | Value |
|-----------|-------|
| First split point (t₀) | 2019-08-01 |
| Step between variants | 2 weeks |
| Test window length | 6 months |

For each variant at time `t`:
- **Training set** — all *Buy* transactions before `t`
- **Test set** — all *Buy* transactions in `[t, t + 6 months)`

**Filtering rules applied per variant:**
1. Only customers who appear in *both* train and test are kept.
2. If a customer already acquired an asset during training, that asset is removed from their test set — the model's job is to recommend *new* assets.

**Interaction matrix** — each variant produces two matrices of shape `(N_customers × 837)`:
- `train_rel_matrix` — acquisition counts during training
- `test_rel_matrix` — acquisition counts during the test window

Both matrices use a globally consistent asset column index (all 837 assets), so models trained on different variants are directly comparable.

All 30 variants are serialised together into `data/processed/datasets.pkl`.

## Models

All models inherit from `BaseRecommender` (`src/models/base.py`) and implement two methods:
- `fit(train_matrix)` — learn from the training interaction matrix
- `recommend(train_matrix, customer_indices, k, exclude_seen=True)` — return top-K ranked asset indices per customer

### Popularity Baseline

Scores each asset by its total acquisition count across all customers (`column sum` of `train_rel_matrix`). Items already held by a customer are masked before ranking.

No learning involved — serves as a strong non-personalised baseline.

### TruncatedSVD (MF)

Applies `sklearn.decomposition.TruncatedSVD` (50 components) to a binarised version of `train_rel_matrix`. The full score matrix is pre-computed as `U Σ Vᵀ` and reused at inference time.

Limitations: optimises reconstruction error rather than ranking loss; treats all zero entries as "score 0", which conflates unseen with disliked.

### BPR Matrix Factorization

A PyTorch re-implementation of the [RecBole BPR model](https://github.com/RUCAIBox/RecBole/blob/master/recbole/model/general_recommender/bpr.py) (Rendle et al., 2009).

| Detail | Value |
|--------|-------|
| Score function | Dot product of user and item embeddings |
| Loss | BPR: `-mean(log(sigmoid(pos_score − neg_score)))` |
| Negative sampling | 1 uniform negative per positive (rejection-sampled) |
| Initialisation | Xavier normal (matches RecBole) |
| Optimiser | Adam, lr = 1e-3 |
| Embedding size | 64 |
| Epochs | 20 |
| Batch size | 2048 |

BPR is preferred over TruncatedSVD for implicit feedback recommendation because its pairwise loss directly optimises ranking, which aligns with how models are evaluated.

## Evaluation

Implemented in `src/evaluation/metrics.py`. Each model is evaluated on all 30 variants and results are averaged.

### Ranking Metric (per-user, then averaged)

| Metric | Description |
|--------|-------------|
| `ndcg@10`, `ndcg@20` | Normalised Discounted Cumulative Gain — rewards placing relevant items early in the list; penalises lower positions via log₂ discounting |

Customers with no test interactions are excluded from the average.

### Beyond-Accuracy Metrics (system-level, computed at K=20)

| Metric | Description |
|--------|-------------|
| `catalog_coverage` | Fraction of the 837-asset catalogue that appears in at least one recommendation list |
| `gini_index` | Concentration of recommendation frequency across items (0 = uniform, 1 = all recommendations on one item) |

These metrics capture whether a model over-concentrates on a small set of popular assets, which is a known failure mode in financial recommendation.
